#### 4. 后端：简单模式性能修复

**问题描述**：使用简单模式提问"你在做什么？"，响应时间超过 13 秒。

**根本原因分析**：
1. 代码侧耗时：3ms（prompt 构建 2ms + 内存写入 1ms）
2. LLM API 耗时：13000+ms（首 token 延迟 13.4 秒）

进一步分析发现，`glm-4.6v` 模型支持 `tool_use`，即使无 worker Agent 也会走 FC 路径，向 LLM 发送大量 tools 定义（`generate_image`, `text_to_speech`, `scrape_webpage` 等），导致 LLM 需要评估所有工具后才能回复。

**解决方案**：无 worker Agent 时跳过 FC 路径，直接走轻量 `_stream_normal_chat`。

**修改文件**：
- `core/chat_engine.py` - 添加 worker 检测，无 worker 时跳过 FC 调度

**核心改动**：
```python
# 路径 3：FC 调度（仅当有员工 Agent 时才启用，避免简单问题走重型 FC 路径）
if _model_supports_tool_use(agent.model_id):
    has_workers = bool(manager.list_workers())
    if has_workers:
        logger.info(f"[路径3-FC] 检测到 {len(manager.list_workers())} 个员工，走 FC 调度")
        async for chunk in _stream_with_fc(...):
            yield chunk
        return
    else:
        logger.debug("[路径3-FC] 无员工 Agent，跳过 FC 调度走轻量路径")
```

**性能提升**：
- 代码侧：3ms → 3ms（无变化）
- 整体：13000ms → 取决于 LLM API 响应（FC 工具评估开销消除）

**诊断日志**：
```
[stream_message] 初始化耗时: 1ms, agent=项目主管, model=glm-4.6v, thinking=False
[normal_chat] prompt构建耗时: 2ms, history_msgs=0
[normal_chat] add_message耗时: 1ms
[normal_chat] 首个chunk耗时: 13412ms (从LLM请求发起到首个响应)
```

结论：99.97% 的时间消耗在 LLM API 响应，代码侧已无优化空间。建议换用响应更快的模型如 `glm-4-flash-250414`。

#### 1. 自定义模型/技能 JSON 并发写入保护

**问题**：`custom_models.json` 和 `custom_skills.json` 的读写无文件锁，并发请求时可能数据丢失或文件损坏。

**修复**：引入 `filelock.FileLock` + `tempfile` 原子写入模式（先写临时文件，再 `os.replace`），与 `MemoryManager` 保持一致。

```python
# api/routes/models.py & api/routes/skills.py
from filelock import FileLock
MODELS_LOCK = FileLock(MODELS_FILE + ".lock", timeout=10)

def _save_custom_models(models: list):
    with MODELS_LOCK:
        fd, tmp_path = tempfile.mkstemp(dir=dir_name, suffix=".tmp")
        try:
            with os.fdopen(fd, "w", encoding="utf-8") as f:
                json.dump(models, f, ensure_ascii=False, indent=2)
            os.replace(tmp_path, MODELS_FILE)
        except Exception:
            if os.path.exists(tmp_path):
                os.unlink(tmp_path)
            raise
```

新增 10 个内置技能，技能总数从 3 个扩展到 13 个。所有技能均支持 FC（Function Calling）自动调度和触发词关键词匹配两种激活方式。78/78 测试通过。

---

### 新增技能列表

#### 1. 代码执行 (`skills/code_execute.py`)

- **功能**：在安全沙盒中执行 Python/JavaScript/Bash 代码
- **安全措施**：
  - 30 秒超时限制
  - 危险代码模式检测（`os.system`、`subprocess`、`rm -rf`、`curl|bash` 等）
  - 临时文件执行，自动清理
  - 输出长度限制（5000 字符）
- **FC 工具名**：`execute_code`
- **触发词**：执行代码、运行代码、跑一下代码、run code 等
- **自动检测**：从用户输入自动识别编程语言，支持 Markdown 代码块提取

#### 2. 翻译 (`skills/translate.py`)

- **功能**：LLM 多语言互译，支持中英日韩法德西俄阿葡等语言
- **特性**：
  - 自动检测源语言和目标语言
  - 保留原文格式（Markdown、换行等）
  - 中文输入默认翻译为英文，反之亦然
- **FC 工具名**：`translate_text`
- **触发词**：翻译、翻译一下、翻译成、translate 等

#### 3. 文档总结 (`skills/doc_summary.py`)

- **功能**：对上传的文档进行结构化分析
- **4 种模式**：
  - `summary`：一句话摘要 + 核心要点 + 详细内容 + 结论
  - `keypoints`：关键信息要点列表
  - `extract`：结构化数据提取（日期、数字、人名等）
  - `analysis`：深度分析（主题、结构、观点、趋势）
- **FC 工具名**：`summarize_document`
- **触发词**：总结文档、帮我总结、归纳一下、提取要点、文档分析 等
- **集成**：复用 `core.file_reader` 读取多种格式

#### 4. 知识库 RAG (`skills/knowledge_base.py`)

- **功能**：上传文档到知识库，基于知识库内容回答问题
- **架构**：
  - 文本分片：按段落 + 句子智能切分（500字/片，50字重叠）
  - 检索引擎：BM25 评分算法（中文 2-gram + 英文单词分词）
  - 问答生成：检索 Top-5 片段 → 注入 LLM context → 生成回答
- **3 种操作**：`add`（添加文档）、`query`（查询）、`list`（列出文档）
- **持久化**：`data/knowledge_base/index.json`
- **FC 工具名**：`knowledge_base_query`
- **触发词**：知识库、添加到知识库、查询知识库 等

#### 5. 数据分析 (`skills/data_analysis.py`)

- **功能**：分析 CSV/Excel/JSON 数据，生成统计报告和可视化图表
- **流程**：
  1. 读取数据文件预览（前 20 行）
  2. LLM 根据预览 + 用户需求生成 pandas + matplotlib 分析代码
  3. 自动执行代码，收集输出和图表文件
- **FC 工具名**：`analyze_data`
- **触发词**：数据分析、统计分析、画图表、柱状图、折线图 等
- **输出目录**：`data/analysis_output/`

#### 6. 定时提醒 (`skills/reminder.py`)

- **功能**：设置定时提醒，支持自然语言时间
- **时间解析**：
  - 相对时间：「5分钟后」「3小时后」「2天后」
  - 绝对时间：「明天9点」「下午3点」「后天上午10点」
  - ISO 格式：`2026-04-06 14:30`
- **后台检查**：daemon 线程每 30 秒检查一次到期提醒
- **持久化**：`data/reminders.json`
- **FC 工具名**：`set_reminder`
- **触发词**：提醒我、设置提醒、闹钟、别忘了 等

#### 7. 搜索报告 (`skills/search_report.py`)

- **功能**：联网搜索并生成结构化调研报告
- **4 种报告格式**：
  - `comprehensive`：综合报告（概述 + 发现 + 详情 + 来源 + 结论）
  - `brief`：简报（3-5 个要点）
  - `comparison`：对比分析（表格呈现）
  - `timeline`：时间线（按时间排列事件）
- **集成**：复用 `core.search` 搜索基础设施
- **FC 工具名**：`generate_search_report`
- **触发词**：搜索报告、帮我调研、搜索并总结 等

#### 8. 邮件发送 (`skills/email_send.py`)

- **功能**：通过 SMTP 发送邮件
- **特性**：
  - LLM 自动润色邮件内容（简短要点 → 正式邮件）
  - 支持附件（上传的文件自动附加）
  - SSL/TLS 加密连接
  - 邮箱格式验证
- **config_schema 配置**：SMTP 服务器、端口、账号、密码/授权码、发件人地址
- **FC 工具名**：`send_email`
- **触发词**：发邮件、发送邮件、写邮件、send email 等

#### 9. 任务管理 (`skills/task_manager.py`)

- **功能**：个人待办事项管理
- **操作**：添加（add）、查看（list）、更新状态（update）、删除（delete）
- **特性**：
  - 优先级：🔴 高 / 🟡 中 / 🟢 低
  - 状态：⬜ 待办 / 🔄 进行中 / ✅ 已完成
  - 自动从用户输入推断操作类型和优先级
  - 支持关键词搜索任务
- **持久化**：`data/user_tasks.json`
- **FC 工具名**：`manage_tasks`
- **触发词**：添加任务、待办、任务列表、完成任务、todo 等

#### 10. 音视频理解 (`skills/media_understand.py`)

- **功能**：音视频语音转文字（ASR）+ LLM 内容分析
- **ASR 支持**：
  - 智谱 AI Whisper API（默认）
  - OpenAI Whisper API
  - 本地 whisper 模型（openai-whisper）
- **视频处理**：自动通过 ffmpeg 提取音频轨道
- **分析输出**：内容摘要 + 关键信息 + 话题分段
- **config_schema 配置**：ASR 服务商、API Key、模型大小
- **支持格式**：MP3/WAV/M4A/FLAC/OGG/MP4/AVI/MKV/MOV 等
- **FC 工具名**：`transcribe_media`
- **触发词**：语音识别、音频转文字、视频转文字、转写 等

## v5.13.1 — 技能安全加固 + 性能优化 + 轻量 FC 模式

### 概述

对 v5.13.0 新增的 10 个技能进行全面代码审查，发现并修复 13 个问题（6 P0 + 5 P1 + 2 P2），新增 3 项性能优化和 1 项新功能。

### P0 — 严重问题修复

#### 1. knowledge_base.py — 索引膨胀 + 并发安全
- **问题**：每个 chunk 的 tokens 数组存入 JSON，导致索引文件膨胀 10 倍+；无文件锁，并发写入可能损坏 index.json
- **修复**：持久化时剔除 tokens（搜索时实时计算）；添加 `_kb_lock` 线程锁；原子写入（tmp+replace）；删除未使用的 `import math`

#### 2. reminder.py — 线程安全 + 内存泄漏
- **问题**：`_load_reminders()` 不持锁修改全局列表与后台线程竞争；已触发/取消的提醒永不清理，无限增长
- **修复**：所有 `_load_reminders()` 调用移入 `_lock` 内；`_save_reminders()` 自动清理超过 30 天的非 pending 提醒；硬限制 200 条上限；原子写入

#### 3. search_report.py — 异步事件循环死锁
- **问题**：`_do_search()` 中 `loop.run_in_executor` + `run_until_complete` 在 FastAPI 运行时死锁
- **修复**：技能 handler 通过 `asyncio.to_thread()` 在独立线程执行，直接使用 `asyncio.run()`

#### 4. data_analysis.py — LLM 生成代码无安全检查
- **问题**：LLM 生成的 Python 分析代码直接执行，无危险模式检测
- **修复**：添加 `_check_code_safety()` 检测 `os.system`、`subprocess`、`exec`、`eval`、`shutil.rmtree` 等危险模式

### P1 — 重要问题修复

#### 5. email_send.py — SMTP 连接泄漏
- **问题**：`server.login()` 失败时 `server.quit()` 不被调用，连接泄漏
- **修复**：`server` 变量提前声明，`finally` 块确保 `server.quit()`；附件增加 20MB 大小限制

#### 6. media_understand.py — 临时文件竞争 + 模型重复加载
- **问题**：`tempfile.mktemp()` 已弃用存在 TOCTOU 竞争；whisper 模型每次调用都重新加载
- **修复**：改用 `NamedTemporaryFile(delete=False)`；全局缓存 `_whisper_model` 避免重复加载

#### 7. translate.py — 触发词过于宽泛
- **问题**：触发词"帮我把"匹配率过高，"帮我把文件删了"会误触翻译技能
- **修复**：从 triggers 和 `_clean_input` 中移除"帮我把"

#### 8. task_manager.py — 并发安全
- **问题**：无文件锁，并发写入可能损坏 user_tasks.json
- **修复**：添加 `_tasks_lock` 线程锁；原子写入（tmp+replace）

#### 9. doc_summary.py — 无文件大小限制
- **问题**：`_read_file_content` 读取超大文件可能 OOM
- **修复**：添加 50MB 文件大小限制（`MAX_FILE_SIZE`）

### 性能优化

#### 10. tool_name → skill O(1) 哈希索引
- **问题**：`execute_tool_by_name` 和 `get_skill_by_tool_name` 每次 O(n) 遍历所有技能
- **修复**：新增 `_TOOL_NAME_INDEX` 字典和 `_rebuild_tool_name_index()` 函数，在 `load_builtin_skills()` 和 `load_custom_skills()` 后自动构建索引

#### 11. 欢迎消息更新
- `get_welcome_message()` 更新为覆盖全部 13 个技能的分类展示

### 新功能

#### 12. 轻量 FC 模式（无 workers 时智能技能调用）
- **背景**：之前无 workers 时跳过 FC 路径，技能只能通过关键词触发
- **改进**：当模型支持 `tool_use` 但无 workers 时，将所有技能 `tool_schema` 注入普通对话路径（`_stream_normal_chat`），LLM 可智能判断并调用技能
- **影响**：用户说"这个 CSV 有什么规律"也能触发数据分析技能，无需精确说出"数据分析"
- **改动**：`_stream_normal_chat` 新增 `tools` 参数；`stream_message` 路径 3 增加无 workers 分支

## v5.14.0 — 龙虾办公室 Canvas 像素艺术 + FC 调度修复（2026-04-06）

将办公室可视化从 CSS/Emoji 方案全面重构为 HTML5 Canvas 像素艺术渲染，实现真正的复古游戏画面效果。同时修复 FC（Function Calling）调度中的关键 Bug：中文员工名导致工具名重复、系统提示词未指示 LLM 使用工具。

### 1. 前端：Canvas 像素艺术渲染器

**问题描述**：原 CSS/Emoji 方案视觉效果差（"跟三岁小孩画的一样"），无法达到游戏级像素艺术效果。

**解决方案**：新建 `PixelCanvas.tsx` 组件，用 HTML5 Canvas 逐像素绘制整个办公室场景。

**创建文件**：
- `frontend/src/components/office/PixelCanvas.tsx` — 533 行 Canvas 渲染器

**核心架构**：

**像素角色特性**：

### 4. 后端：FC 调度 — 中文员工工具名修复

**问题描述**：中文员工名经 `re.sub(r'[^a-zA-Z0-9_]', '', name)` 处理后变为空字符串，所有员工工具名均为 `assign_to_____`，LLM 无法区分。

**日志证据**：
```
[FC调度] 工具列表: ['assign_to_____', 'assign_to______', 'assign_to_____', 'assign_to_____', 'assign_to_____']
```

**修复文件**：`core/chat_engine.py` — `_stream_with_fc` 函数

**修复内容**：
```python
# 修复前
safe_name = re.sub(r'[^a-zA-Z0-9_]', '_', w.name).lower()

# 修复后
safe_name = re.sub(r'[^a-zA-Z0-9_]', '', w.name).lower()
if not safe_name:
    safe_name = f"worker_{idx + 1}"  # 中文名回退为 worker_N
# 去重
existing_names = {d["function"]["name"] for d in worker_tool_definitions}
candidate = f"assign_to_{safe_name}"
if candidate in existing_names:
    candidate = f"assign_to_{safe_name}_{idx + 1}"
```

**修复后工具名示例**：
```
['assign_to_worker_1', 'assign_to_worker_2', 'assign_to_worker_3', 'assign_to_worker_4', 'assign_to_worker_5']
```
