> **更新日期：** 2026-04-05

## 一、项目简介

GymClaw 是一个对话式的 AI 助手系统。用户只需要一个聊天框，就能通过自然语言完成各种任务——从日常问答、主题调研、图片生成到文件分析，全部在一个界面内完成。

### 核心特点

- **会话隔离**：`streamingSessionId` 机制防止跨会话消息泄漏，切换会话时立即清空 messages
- **标题实时更新**：发送首条消息时立即用 snippet 设置临时标题，AI 生成后刷新会话列表
- **模型 ID 解析**：`resolve_model_id()` 统一映射 agent 配置的模型 ID 到 provider 实际模型 ID
- **GLM-4.7-Flash 思考切换**：同一模型通过 `enable_thinking` 参数切换思考/简单模式（强制思考 vs 自动判断）
- **深色模式**：支持浅色/深色/跟随系统三种主题，自动持久化到 localStorage
- **Toast 通知系统**：React Context + useToast Hook 实现，支持 success/error/info 三种类型，自动消失（默认 3 秒），点击关闭，深色模式适配，右侧滑入动画，Z-index 9999，响应式宽度（280px-400px）
- **错误边界**：React ErrorBoundary 组件捕获子组件渲染错误，防止整页白屏，显示友好错误提示
- **API 速率限制**：基于 slowapi 的限流保护（默认 60次/分钟）
- **并发安全**：MemoryManager 移除 `_current_session` 共享状态，filelock 保护 JSON 写入
- **性能优化**：ConfigLoader 单例缓存、React.memo 减少重渲染、Zustand slice 模式重构
- **性能优化**：httpx 连接池复用、字符串拼接 O(n) 优化、配置单次解析、内存缓存（MemoryManager 30s TTL + LLM 实例缓存 300s TTL）、asyncio.Lock 防并发泄漏
- **安全加固**：SSRF 防护（`is_safe_url`）、API 错误体脱敏、agent_id 格式校验
- **代码质量**：模块拆分（security/model_router/prompt_builder/errors）、pytest 21项测试覆盖
- **自定义模型编辑**：支持编辑自定义模型的全部字段（名称/供应商/模型ID/Key/能力等）
- **Agent 模型可搜索选择**：Agent 编辑表单中模型选择改为可搜索下拉，自动匹配供应商，展示模型描述、能力徽章、思考模型标记（🧠）
- **模型描述系统**：内置模型自带描述说明（如"智谱 AI 语言模型，128K 上下文"），自定义模型新增描述栏位，模型列表和 Agent 选择框均展示描述
- **流式输出优化**：绕过 Next.js 代理缓冲，开发模式前端直连后端 SSE 端点
- **API Key 安全**：自定义模型 Key 加密存储（SHA-256+XOR+Base64），列表展示脱敏，.gitignore 排除敏感文件
- **思考过程持久化**：AI 思考过程随消息保存，历史对话可随时查看
- **清空全部对话**：侧边栏底部提供清空按钮，一键删除所有会话，带确认弹窗
- **零门槛**：打开网页即可使用，预置模型和默认 Agent
- **多员工架构**：一个主管（默认 Agent）+ 多个员工 Agent，支持 @指派和主管自动调度
- **智能员工类型**：系统自动推导员工类型——执行型（⚡ Runner，直接调用技能，零 LLM 消耗）、混合型（🧠 Smart，LLM + FC 工具）、智能型（💬 Agent，纯 LLM 创作）
- **技能配置系统**：每个技能通过 `config_schema` 声明配置需求（text/password/select/number/boolean），用户在独立技能管理页面配置 API Key、模型、参数等，配置持久化到 `data/skill_configs.json`
- **独立技能管理页**：设置页新增「技能管理」Tab，展示所有技能卡片（内置/自定义），支持启用/禁用、编辑配置（动态表单渲染）、查看配置状态（✅已配置/❌未配置）
- **技能绑定**：创建/编辑员工时可绑定技能，模型和技能均为可选，系统根据配置自动判断执行方式，绑定 UI 显示技能配置状态（🟢/🔴）
- **技能智能分配**：主管只拿到未被员工绑定的技能工具，有对应技能的员工优先被调度
- **参与信息展示**：消息中显示参与的 Agent 列表（主管/员工），方便追踪任务执行链路
- **FC 自动检测**：测试模型连接时自动检测 Function Calling 支持
- **能力精简**：只保留 tool_use 能力标签，去掉 text/vision/image_gen 等无实际逻辑的标签
- **技能转工具**：技能不再用户可见，转为内部工具（tool_schema），由主管 Agent 通过 FC 自动调用
- **智能路由**：上传文件时自动检测模型能力，切换到合适的模型
- **深度思考**：Agent 可启用深度思考模式（`enable_thinking`），自动切换到推理模型（如 GLM-Z1-Flash、DeepSeek-R1）；简单模式下所有路径（普通对话、FC 调度、员工任务、技能匹配）均跳过思考模型切换并过滤思考内容
- **联网搜索**：全局搜索配置（一个搜索引擎、一个 API Key），对话时独立开关；支持双路径搜索（keys 匹配时原生 web_search 工具，否则独立搜索 API + 上下文注入）
- **模型信息中心**：`core/model_info.py` 统一管理模型查询，30 秒缓存避免重复加载
- **多语言**：界面支持中文/英文切换
- **流式输出**：SSE 实时流式显示 AI 回复
- **思考过程**：支持展示模型思考过程（DeepSeek R1 等）
- **智能标题**：AI 根据对话内容自动生成简短标题
- **管道可视化**：流式输出时展示当前 Agent、模型、技能信息
- **文件附件**：消息中展示上传的文件，支持模态框内预览（图片/视频/音频/表格/代码/文档）
- **文件持久化**：上传文件信息随消息一起保存，切换对话后附件不丢失
- **欢迎页增强**：WelcomeScreen 支持联网搜索开关、简单/思考模式切换，默认选择"思考"模式，与对话内部 InputBar 功能一致
- **默认思考模式**：chatStore 的 chatMode 默认值改为 `"think"`，所有界面默认启用思考模式
- **自动滚动**：切换对话后自动滚动到最新消息

## 四、核心模块说明

### 4.1 对话引擎 (`core/chat_engine.py`)

对话引擎是整个系统的核心，负责：

1. **@指派**：解析 @员工名，直接指派给对应员工 Agent（Runner 直接执行技能，Smart/Agent 走 LLM）
2. **FC 调度**：主管 Agent 通过 Function Calling 自动调度**员工**（员工被包装为 FC tool：`assign_to_{name}`）
   - **Runner（执行型）**：直接调用绑定的技能工具，跳过 LLM，零 token 消耗
   - **Smart（混合型）**：LLM 思考后通过 FC 调用绑定的技能工具
   - **Agent（智能型）**：纯 LLM 对话/创作
3. **技能智能分配**：主管只拿到未被任何员工绑定的技能工具，有对应技能的员工优先被调度
4. **深度思考切换**：`_resolve_thinking_agent()` 在调度员工时自动检查 `enable_thinking`，如当前模型不支持思考则自动切换到推理模型。简单模式下（`enable_thinking=False`）所有路径均跳过此切换
5. **思考内容过滤**：`_stream_llm_real()` 的 `enable_thinking` 参数控制 `reasoning_content` 和 `<think...</think` 标签内容的输出——简单模式下思考内容被静默丢弃，不发送到前端
6. **双路径搜索**：联网搜索采用全局配置，支持两种执行路径：
   - **Path A**：全局搜索 Key == 当前 Agent Key → 注入原生 `web_search` 工具（零额外 API 调用）
   - **Path B**：Keys 不同 → 调用独立搜索 API → 将结果注入对话上下文 → 调用模型
7. **关键词匹配**：优先匹配当前 Agent 绑定的技能，无匹配则全局匹配
8. **流式输出**：通过 SSE 实时推送 AI 回复
9. **思考过程**：支持展示模型的思考过程（如 DeepSeek R1、GLM-Z1-Flash）
10. **智能标题**：首次对话后调用 LLM 生成简短标题
11. **参与信息**：done 事件携带 `agents` 列表，消息中展示参与的 Agent

**处理流程（5 条执行路径）：**

```
用户消息 + 文件
     │
     ▼
路径 1: @指派 → 解析 @员工名 → 直接指派给对应员工 Agent
     │
     ▼（无 @）
路径 2: 安全检查 → 敏感词/注入检测
     │
     ▼（安全通过）
路径 3: FC 调度 → 主管 Agent 通过 Function Calling 自动调度
     │               │
     │               ├── 员工调度 → assign_to_{name} → 员工使用绑定的模型+技能处理
     │               └── 工具调度 → 调用内部工具（图片生成、TTS 等）
     │
     ▼（FC 不可用或未触发）
路径 4: 关键词匹配 → 按触发词匹配内置技能 → 调用技能 handler
     │
     ▼（无匹配）
路径 5: 普通对话 → 使用主管模型直接对话
```

**能力检测规则：**

| 文件类型 | 扩展名 | 所需能力 |
|---------|--------|---------|
| 图片 | .png, .jpg, .jpeg, .gif, .webp, .bmp, .tiff, .tif, .svg, .ico | `vision` |
| 表格 | .xls, .xlsx, .csv | `spreadsheet` |
| 文档 | .pdf, .doc, .docx | `text` |
| 代码/文本 | .py, .js, .ts, .html, .css, .vue, .jsx, .tsx 等 | `text` |
| 无文件 | — | `text` |

### 4.2 LLM 工厂 (`core/llm_factory.py`)

负责创建和管理 LLM 实例：

- **统一接口**：通过 CrewAI LLM 类统一调用不同供应商的模型
- **缓存机制**：相同参数的 LLM 实例会被缓存，避免重复创建
- **联网搜索**：支持配置 `internet_search` 参数
- **自定义 API**：支持自定义 base_url 和 api_key

### 4.3 技能系统 (`skills/`)

每个技能是一个 Python 模块，包含：

| 组件 | 说明 |
|------|------|
| `SKILL_META` | 技能元数据（名称、图标、触发词、描述） |
| `config_schema` | 配置声明（可选），定义技能需要的配置项（API Key、模型选择、参数等） |
| `handle()` | 技能处理函数，接收消息和上下文，返回回复 |

**技能配置系统（config_schema）：**

每个技能可通过 `@register_skill` 装饰器的 `config_schema` 参数声明配置需求：

| 配置类型 | 说明 | 示例 |
|---------|------|------|
| `text` | 文本输入 | 自定义参数 |
| `password` | 密码输入 | API Key |
| `select` | 下拉选择 | 模型选择、图片尺寸 |
| `number` | 数字输入 | 超时时间 |
| `boolean` | 开关 | 是否启用某功能 |

配置优先级：`data/skill_configs.json` > 环境变量（`env_hint`） > `default` 默认值。

配置状态（config_status）：
- `"ok"`：所有必填项已配置
- `"missing"`：存在未配置的必填项
- `"none"`：无必填配置项

**内置技能列表：**

| 技能 ID | 名称 | 触发词 | 描述 |
|---------|------|--------|------|
| `image_generate` | 图片生成 | 画、生成图片、绘制 | 使用 CogView-4 生成图片 |
| `tts` | 语音合成 | 语音、转语音、合成 | 使用 Edge-TTS 生成语音 |
| `web_scrape` | 网页抓取 | 抓取、爬取、抓网页 | 抓取网页内容 |

**技能匹配算法：**

```python
# 遍历所有已启用技能
# 检查触发词是否出现在用户输入中（不区分大小写）
# 匹配分数 = 触发词长度（越长越精确）
# 反向匹配（输入包含在触发词中）得分较低
# 选分数最高的技能执行
# 无匹配则走通用对话
```

**自定义技能：**

自定义技能保存到 `data/custom_skills.json`，支持两种类型：
- **Prompt 模板**：定义系统提示词，AI 按模板回答
- **外部 API**：调用外部 API 获取结果

### 4.4 Agent 管理 (`core/agents.py`)

- 支持创建多个 Agent，每个有独立的名称、模型、温度、**技能**、**深度思考**配置
- 默认 Agent 名称为 "GymClaw"，头像为 "GC"
- 默认 Agent 不可删除
- 对话中通过 `@名字` 切换 Agent
- **技能绑定**：每个 Agent 可绑定多个技能（如 `image_generate`、`tts`），技能决定员工的处理能力
- **深度思考开关**：每个 Agent 可启用 `enable_thinking`，启用后自动切换到推理模型处理

- 主管 Agent 通过 FC 将任务调度到绑定了对应技能的员工
- Agent 配置持久化到 `data/agents.json`

### 4.5 会话记忆 (`core/memory.py`)

- 每个会话独立的消息历史
- `add_message` 支持 `metadata` 参数，可保存 `files`、`thinking` 等额外字段
- 消息超过阈值时自动生成摘要，减少 Token 消耗
- 数据持久化到 `data/memory/` 目录（`{session_id}.json`）
- 会话元数据（标题、置顶状态）存储在 `data/sessions/` 目录
- **双数据源合并**：`api/routes/sessions.py` 同时读取 `data/sessions/` 和 `data/memory/`，合并返回
- **Session ID 一致性**：`_resolve_session` 使用前端传入的 UUID，不自动生成 `sess_*` 格式

### 4.6 配置加载器 (`core/config_loader.py`)

- 从 `config/config.yaml` 加载系统配置
- 支持 `${ENV_VAR}` 语法替换环境变量
- 配置内容：LLM 供应商、模型列表、工具注册、Agent 模板

### 8.3 FC 自动检测与智能路由


| 路径 | 条件 | 执行方式 |

## 十二、常见问题

### Q: 启动后提示 API Key 错误？

确保 `.env` 文件中已配置 `ZHIPU_API_KEY`，或在前端设置页面添加自定义模型并填入 API Key。

### Q: 前端连接不上后端？

确认后端已在 8000 端口启动。检查 `frontend/src/lib/api.ts` 中的 `API_BASE` 配置。

### Q: 上传图片后没有分析？

确保系统中有支持 `vision` 能力的模型（如 GLM-4V-Flash）。智能路由会自动切换。

### Q: 上传的文件类型不支持？

系统支持 70+ 种文件类型，包括所有主流前端/后端代码文件、配置文件、图片、视频、音频、文档等。如果仍然不支持，可以在 `api/routes/files.py` 的 `ALLOWED_EXTENSIONS` 中添加。

### Q: 切换对话后文件附件消失？

v4.1.0 已修复此问题。文件信息会随消息一起保存到后端 memory 中，切换对话后附件正常显示。

### Q: 如何添加 DeepSeek 等其他模型？

在前端「设置 - 模型配置」中点击「添加模型」，填入：
- 模型名称：DeepSeek R1
- API Base URL：https://api.deepseek.com
- API Key：你的 Key
- 模型 ID：deepseek-reasoner
- 选择能力标签（文本、视觉等）

### Q: 自定义技能如何使用？

在前端「设置 - 技能管理」中点击「添加技能」，填入名称、触发词和执行提示词。保存后，在对话中输入包含触发词的消息即可触发。
